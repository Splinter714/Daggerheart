MOTIVES & TACTICS ALIGNMENT DEBUG PROBLEM
==========================================

PROBLEM DESCRIPTION:
The "Motives & Tactics" section of the expanded adversary card has a subtle vertical shift between view and edit modes. The text within the bordered boxes appears at slightly different vertical positions. Both modes use identical styling (including padding, height, line-height, and border) but the text baseline/positioning differs, causing a visual jump when toggling between view and edit. The goal is to eliminate this shift while keeping the label and the text content/input on the same line.

CURRENT STATE:
- Using flexbox with align-items: baseline for containers
- Both view and edit modes have identical height (1.8rem), padding (0.25rem), line-height (1.2)
- Both have identical borders (1px solid var(--border))
- Text still shifts vertically between modes despite identical styling

WHAT'S BEEN TRIED:
=================

1. TRANSPARENT BORDER STRATEGY:
   - Added transparent borders to view mode to match edit mode borders
   - This worked for name and description sections but not motives

2. FLEXBOX APPROACHES:
   - align-items: center (centered vertically but caused other issues)
   - align-items: baseline (current approach)
   - align-items: flex-start (didn't help)

3. VERTICAL ALIGNMENT:
   - vertical-align: top
   - vertical-align: middle
   - vertical-align: baseline

4. HEIGHT AND LINE-HEIGHT MATCHING:
   - Both containers: height: 1.8rem
   - Both content/input: height: 1.8rem, line-height: 1.2
   - Both labels: height: 1.8rem, line-height: 1.8rem

5. PADDING AND MARGIN CONTROL:
   - Both content/input: padding: 0.25rem
   - Both content/input: margin: 0
   - Both containers: padding: 0

6. CSS SPECIFICITY:
   - Added !important to override App.css rules
   - Increased selector specificity

7. DISPLAY PROPERTIES:
   - display: flex (current)
   - display: inline-block (caused wrapping issues)
   - display: inline (didn't work)

8. DEBUGGING TECHNIQUES:
   - Added colored borders to visualize element boundaries
   - Added background fills to reveal invisible space
   - Temporarily removed save/cancel buttons to isolate issue

CURRENT ISSUE:
==============

Despite having identical styling properties, the motives section still has a subtle vertical shift. The text appears to be positioned differently within the bordered containers between view and edit modes. This suggests there might be:

1. Browser default styling differences between <span> and <input> elements
2. Font rendering differences between text content and input fields
3. Some CSS property not being applied consistently
4. A layout calculation difference between flex containers

QUESTIONS FOR DEBUGGING:
========================

1. Are there any browser default styles affecting <input> vs <span> that we haven't overridden?
2. Is there a difference in how browsers calculate text baseline for input fields vs text content?
3. Should we try matching the exact approach used in the working name section?
4. Are there any CSS properties we're missing that could cause this shift?

The goal is to achieve pixel-perfect alignment like the name and description sections.
